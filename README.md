# Книгоманы

**Книгоманы** — веб-сайт для любителей книг, созданный как дипломный проект. Платформа помогает находить книги, читать и писать рецензии, вести личную книжную полку, общаться на форуме и поддерживать порядок в сообществе через систему модерации.

Проект ориентирован на понятный пользовательский интерфейс, разделение ролей и удобное администрирование контента.

## О проекте

Сайт объединяет каталог книг, личный кабинет читателя, рецензии, форум и административную панель. Пользователь может найти книгу, добавить её на свою полку, оставить рецензию, отслеживать уведомления и обсуждать темы на форуме.

Администраторы и модераторы получают отдельную панель управления. Администратор отвечает за книги, жанры, пользователей, статистику и импорт, а модератор — за проверку рецензий и сообщений форума.

## Быстрый старт

### 1. Клонирование репозитория

```bash
git clone https://github.com/SetchatkaGlaza/Diplom-Website-for-book-lovers.git
cd Diplom-Website-for-book-lovers
```

### 2. Установка зависимостей

```bash
npm install
```

### 3. Настройка окружения

Создайте файл `.env` в корне проекта и заполните переменные окружения. Минимальный пример находится в разделе [Переменные окружения](#переменные-окружения).

### 4. Подготовка базы данных

Создайте PostgreSQL-базу данных и укажите доступы к ней в `.env`.

Для локальной разработки можно включить автоматическую синхронизацию схемы:

```env
DB_SYNC_ALTER=true
```

### 5. Запуск приложения

```bash
npm start
```

После запуска сайт будет доступен по адресу:

```text
http://localhost:3000
```

Если переменная `PORT` задана, приложение запустится на указанном порту.

## Переменные окружения

Пример `.env` для локального запуска:

```env
NODE_ENV=development
PORT=3000
SESSION_SECRET=replace_with_long_random_secret

DB_NAME=book_lovers
DB_USER=postgres
DB_PASSWORD=password
DB_HOST=localhost
DB_PORT=5432
DB_DIALECT=postgres
DB_LOGGING=false
DB_SYNC_ALTER=true

CLOUDINARY_CLOUD_NAME=your_cloud_name
CLOUDINARY_API_KEY=your_api_key
CLOUDINARY_API_SECRET=your_api_secret

EMAIL_HOST=smtp.example.com
EMAIL_PORT=587
EMAIL_USER=example@example.com
EMAIL_PASS=password
EMAIL_FROM=example@example.com
```

Также приложение поддерживает подключение к базе через одну строку:

```env
DATABASE_URL=postgres://user:password@host:5432/database
```

Если задан `DATABASE_URL` или `DB_URL`, он будет использован вместо отдельных `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST` и `DB_PORT`.

## Работа с базой данных

Проект использует Sequelize и PostgreSQL. Модели находятся в папке `models/`, а связи между ними описаны в `models/associations.js`.

Основные сущности:

- `User` — пользователи и роли;
- `Book` — книги каталога;
- `Genre` — жанры;
- `Review` — рецензии;
- `UserBook` — личные книжные полки;
- `Notification` — уведомления;
- `ForumCategory` — категории форума;
- `ForumTopic` — темы форума;
- `ForumPost` — сообщения форума;
- `ForumPostModeration` — записи модерации и апелляций.

Для разработки можно использовать `DB_SYNC_ALTER=true`, чтобы Sequelize обновлял схему автоматически. Для production-среды лучше применять миграции и не включать автоматическое изменение схемы.

## Cloudinary

Cloudinary используется для хранения аватарок пользователей и обложек книг.

### Необходимые переменные

```env
CLOUDINARY_CLOUD_NAME=your_cloud_name
CLOUDINARY_API_KEY=your_api_key
CLOUDINARY_API_SECRET=your_api_secret
```

### Как работает загрузка

- Multer принимает файл в память через `memoryStorage`.
- Sharp подготавливает изображение: изменяет размер и формат.
- Cloudinary SDK отправляет изображение signed-запросом.
- В базе сохраняются URL изображения и `public_id`.
- При замене изображения старый файл может быть удалён из Cloudinary по `public_id`.

### Частые причины ошибок Cloudinary

- неверный `cloud_name`;
- неверные `api_key` или `api_secret`;
- пробелы или лишние кавычки в `.env`;
- заблокированный или исчерпавший лимиты Cloudinary-аккаунт;
- неподдерживаемый тип файла.

## Проверка проекта

В проекте пока нет отдельного npm-скрипта для тестов. Для базовой проверки можно использовать следующие команды.

### Проверка синтаксиса JavaScript

```bash
for f in $(rg --files -g '*.js' -g '!node_modules'); do node -c "$f" || exit 1; done
```

### Проверка EJS-шаблонов

```bash
node - <<'NODE'
const { execSync } = require('child_process');
const fs = require('fs');
const ejs = require('ejs');

const files = execSync('rg --files -g "*.ejs"', { encoding: 'utf8' })
  .trim()
  .split('\n')
  .filter(Boolean);

for (const file of files) {
  ejs.compile(fs.readFileSync(file, 'utf8'), { filename: file });
}

console.log(`Compiled ${files.length} EJS files`);
NODE
```

### Проверка пробелов и конфликтных символов в diff

```bash
git diff --check
```

## Production-рекомендации

Перед публикацией сайта рекомендуется:

- задать `NODE_ENV=production`;
- использовать длинный случайный `SESSION_SECRET`;
- подключить постоянное хранилище сессий вместо стандартного MemoryStore;
- хранить секреты только в переменных окружения;
- отключить `DB_SYNC_ALTER`;
- использовать миграции для изменения схемы базы данных;
- включить HTTPS;
- настроить регулярные резервные копии базы данных;
- проверить лимиты Cloudinary;
- ограничить доступ к административным аккаунтам надёжными паролями.
